# Утверждение архитектуры приложения PromptMaster (MVP)

Дата: 2026-04-02  
Версия: 1.0 (MVP)

## Подход и основные принципы

Для MVP выбран **монолитный подход**: вся функциональность бота (обработка Telegram-взаимодействий, генерация промптов ИИ, платежи и доступ к данным) работает **в рамках одного приложения/процесса**.  
Модульность обеспечивается за счёт **внутренней декомпозиции на пакеты/сервисы**, а не через микросервисы.

Цели монолита для MVP:
- быстрее собрать работающий продукт за срок диплома;
- минимизировать сложность инфраструктуры (один сервис, одна схема деплоя);
- обеспечить высокий `code reuse` между частями (ядро генерации промптов и Telegram-слой).

## Общий поток запроса пользователя

1. Пользователь отправляет в Telegram команду или сообщение.
2. `Telegram bot` маршрутизирует событие в сценарий генерации (MVP: мастер “3 вопроса”).
3. `AI module` формирует итоговый промпт (включая системную роль и логику итеративного улучшения) и запрашивает LLM.
4. Сценарий проверяет доступ пользователя:
   - если доступ бесплатный — генерация выполняется сразу;
   - если нужен premium — до генерации инициируется платеж/проверка статуса.
5. `DB module` хранит пользователей, промпты и платежи; после подтверждения платежа снимает ограничения.

## Основные модули

### 1) `Telegram bot` (обработка команд и сообщений)

Ответственности:
- Обработка входящих апдейтов Telegram: команды (`/start`, `/help`), сообщения, callback’и.
- Сценарии общения:
  - “мастер промптов” (MVP): последовательность из 3 вопросов;
  - ветвления по типу запроса (в MVP — базовый текстовый сценарий; расширения под `image/audio/video` — планируются).
- Контроль ограничений:
  - проверка статуса доступа (free/premium);
  - простые лимиты/антифлуд (при необходимости через Redis).
- Интеграция с другими модулями:
  - вызывает `AI module` после формирования входных данных и проверки доступа;
  - вызывает `Payments module` для создания/проверки платежей;
  - читает/пишет изменения в `DB module`.

Технологии:
- `Python 3.12`
- `aiogram 3.x` + `asyncio`

### 2) `Модуль ИИ` (генерация промптов через GPT 4o mini)

Ответственности:
- Сборка “финального промпта” по данным пользователя:
  - системная роль (system prompt);
  - структура промпта (контекст, цель, ограничения);
  - правила формата ответа (например: план + готовый текст/шаблон).
- Вызов LLM и получение результата.
- Итеративное улучшение (self-refine loop в MVP допускается как 1..N повторных прогонов; количество шагов фиксируется для контроля стоимости).
- Нормализация результата:
  - фильтрация служебного мусора;
  - приведение к требуемому формату (для копирования/использования пользователем).
- Кэширование промежуточных результатов (опционально): чтобы снижать стоимость и ускорять ответы для повторов.

Технологическая договорённость для MVP:
- Модель: **GPT-4o mini** (через OpenAI API).

Интерфейсы (внутренние):
- `generate_prompt(user_input, user_profile, options) -> PromptResult`

### 3) `Модуль платежей` (интеграция с ЮKassa)

Ответственности:
- Создание платежей по действию пользователя (например, “разблокировать премиум”).
- Обработка статусов оплаты (подтверждение/отмена/ошибка):
  - обработчик webhook’а/колбэков ЮKassa (в зависимости от выбранного режима);
  - обновление статуса в БД и разблокировка функций.
- Поддержка сценариев MVP:
  - freemium-модель (часть функций доступна без оплаты);
  - разблокировка после успешной оплаты.

Данные, которые хранит/использует модуль:
- идентификатор платежа в ЮKassa;
- связка `telegram_user_id` ↔ `payment_id` ↔ статус оплаты;
- дата/сумма/валюта и срок действия premium (если премиум временной).

Внутренний интерфейс:
- `create_payment(user_id, amount, product) -> PaymentLink/PaymentId`
- `handle_payment_event(event_payload) -> PaymentUpdateResult`

### 4) `Модуль БД` (хранение пользователей, промптов, платежей)

Ответственности:
- Хранение и чтение:
  - пользователей (id Telegram, профиль/настройки, статус доступа);
  - сгенерированных промптов (в MVP — минимум история последнего/последних промптов, при необходимости расширить);
  - платежей (платежи, статусы, связи с пользователями).
- Обеспечение целостности данных:
  - уникальные ключи (например, по `telegram_user_id` и/или внешним `payment_id`);
  - транзакционность при изменении статуса “доступ premium”.
- Репозитории/DAO для изоляции SQL от бизнес-логики.

Хранилища для MVP:
- `SQLite` (минимально для старта) или `PostgreSQL` (если планируется сразу прод-готовность).

Дополнительно:
- `Redis` может использоваться для кеша/лимитов (необязательная часть MVP).

Минимальные сущности (логическая модель):
- `users` (telegram_user_id, name/locale, access_level, created_at, updated_at)
- `prompts` (prompt_id, telegram_user_id, input_payload, prompt_text, model, created_at)
- `payments` (payment_id, telegram_user_id, yookassa_payment_id, amount, status, created_at, updated_at)

## Взаимодействие модулей (внутри монолита)

Внутри одного приложения модули общаются через чёткие внутренние сервисные интерфейсы:
- `Telegram bot`:
  - получает событие → вызывает `DB module` для проверки доступа;
  - при необходимости инициирует платеж через `Payments module`;
  - после подтверждения/при free-доступе вызывает `AI module` и отправляет результат.
- `Payments module`:
  - создаёт платеж и фиксирует сущности в БД;
  - при событии от ЮKassa обновляет БД и переводит пользователя в premium.
- `AI module`:
  - использует `DB module` (например, профиль пользователя) и возвращает готовый промпт;
  - фиксирует итог в `prompts`.

## Технологический стек (по ТЗ)

- `Python 3.12`, `aiogram 3.x`, `asyncio`
- `OpenAI API` (для GPT-4o mini)
- `SQLite/PostgreSQL`, `Redis` (cache/limits)
- `pytest` (ориентир: 80% покрытия для core функций)
- `Docker` / `Docker Compose`
- Наблюдаемость: `Sentry` (в ТЗ помечено как TODO)

## Расширяемость после MVP

Монолитный подход не мешает расширяться: новые сценарии (`/image`, `/audio`, `/video`) добавляются как новые обработчики в `Telegram bot` и новые ветки в `AI module`, при сохранении общей БД-схемы и интерфейсов доступа.
